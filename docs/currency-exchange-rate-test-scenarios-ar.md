# سيناريوهات اختبار نظام العملات وأسعار الصرف والتكامل المالي

## المتطلبات المشتركة
- مستخدم مسجل دخول عبر Sanctum.
- إرسال ترويسة `X-Company-Id` للشركة الحالية.
- وجود سنة مالية مفتوحة، ودفتر يومية نشط، وحسابين قابلين للترحيل على الأقل.
- وجود عملة أساسية للشركة، مثل ILS أو USD، وعملات نشطة أخرى مثل USD وEUR وAED.
- عند اختبار مصدر السوق، يجب ضبط `OPEN_EXCHANGE_RATES_APP_ID` في `.env`.

## 1. فحص قائمة العملات للشركة
الخطوات:
1. نفذ `GET /api/core/currencies`.
2. تحقق من رجوع العملات العالمية وعملات الشركة فقط.
3. تحقق من وجود الحقول: `rate_source`, `rate_source_label`, `exchange_rate`, `effective_rate`, `company_rate`, `has_override`, `override_date`, `is_company_base_currency`.

النتيجة المتوقعة:
- العملة الأساسية للشركة تعرض `effective_rate = 1`.
- العملات الأخرى تعرض سعرها الفعلي حسب مصدر الشركة الحالي.

## 2. فحص Dropdown العملات للقيود والفواتير
الخطوات:
1. نفذ `GET /api/core/currencies/dropdown`.
2. تحقق من ظهور العملات النشطة فقط.
3. تحقق أن كل عملة تحمل السعر الفعلي المطلوب للاستخدام في الواجهات.

النتيجة المتوقعة:
- القائمة صالحة لبناء حقول اختيار العملة في القيود والفواتير.

## 3. مصدر سعر النظام System Rate
الخطوات:
1. نفذ `PUT /api/core/currencies/source` مع body: `{ "source": "system" }`.
2. نفذ `GET /api/core/currencies/source`.
3. نفذ `GET /api/core/currencies/{currency_ulid}/rate` لعملة غير أساسية.

النتيجة المتوقعة:
- `can_edit_rates = false`.
- `can_sync = false`.
- السعر الفعلي يأتي من `currencies.exchange_rate` نسبة إلى عملة الشركة الأساسية.
- محاولة `POST /api/core/currencies/{currency_ulid}/rate` يجب أن تفشل برسالة تفيد بضرورة تبديل المصدر إلى سعر الشركة أولاً.

## 4. مصدر سعر الشركة Company Rate
الخطوات:
1. نفذ `PUT /api/core/currencies/source` مع body: `{ "source": "manual" }`.
2. اختر عملة غير أساسية، ثم نفذ `POST /api/core/currencies/{currency_ulid}/rate` مع body: `{ "rate": 3.72000000, "date": "2026-05-04" }`.
3. نفذ `GET /api/core/currencies/{currency_ulid}/rate`.
4. نفذ `GET /api/core/currencies`.

النتيجة المتوقعة:
- `source = manual` و `rate_source_label = Company Rate`.
- `company_rate = 3.72`.
- `has_override = true`.
- `effective_rate = company_rate`.
- يظهر `override_date = 2026-05-04`.

## 5. حذف سعر الشركة والرجوع لسعر النظام
الخطوات:
1. بعد إنشاء سعر شركة خاص، نفذ `DELETE /api/core/currencies/{currency_ulid}/rate`.
2. نفذ `GET /api/core/currencies/{currency_ulid}/rate`.

النتيجة المتوقعة:
- يتم حذف override الخاص بالشركة soft delete.
- `has_override = false`.
- `effective_rate` يعود إلى سعر النظام.

## 6. مصدر سعر السوق Market API Rate
الخطوات:
1. تأكد من وجود `OPEN_EXCHANGE_RATES_APP_ID`.
2. نفذ `PUT /api/core/currencies/source` مع body: `{ "source": "market" }`.
3. نفذ `POST /api/core/currencies/sync`.
4. نفذ `GET /api/core/currencies`.

النتيجة المتوقعة:
- `can_sync = true`.
- يتم إنشاء أو تحديث سجلات في `company_exchange_rates` بالمصدر `market`.
- تظهر أسعار السوق كسعر فعلي للشركة.
- عند غياب API key يجب رجوع خطأ واضح بدون كسر النظام.

## 7. تحويل مبلغ بين عملتين
الخطوات:
1. نفذ `POST /api/core/currencies/convert` مع body:
   ```json
   {
     "amount": 100,
     "from_currency_ulid": "FROM_ULID",
     "to_currency_ulid": "TO_ULID",
     "date": "2026-05-04"
   }
   ```
2. جرّب التحويل من العملة الأساسية إلى عملة أجنبية.
3. جرّب التحويل من عملة أجنبية إلى العملة الأساسية.
4. جرّب التحويل بين عملتين أجنبيتين.

النتيجة المتوقعة:
- `converted_amount` محسوب حسب مصدر سعر الصرف الحالي.
- `rate` يعكس معدل التحويل النهائي بين العملتين.
- التحويل من العملة إلى نفسها يرجع نفس المبلغ وسعر `1`.

## 8. إنشاء قيد بعملة الشركة الأساسية
الخطوات:
1. نفذ `POST /api/financial/entries` بسطرين متوازنين بدون `currency_id`:
   - مدين 100 على حساب أصول.
   - دائن 100 على حساب آخر.
2. نفذ `GET /api/financial/entries/{entry_ulid}/show`.

النتيجة المتوقعة:
- يتم استخدام عملة الشركة الأساسية تلقائياً.
- `exchange_rate = 1`.
- `debit_original/credit_original` تساوي مبالغ العملة الأساسية.
- القيد متوازن.

## 9. إنشاء قيد بعملة أجنبية مع احتساب تلقائي للسعر
الخطوات:
1. تأكد أن مصدر السعر مضبوط وأن للعملة الأجنبية سعر فعال.
2. نفذ `POST /api/financial/entries` بسطرين:
   - السطر الأول: `currency_ulid = USD_ULID`, `debit_original = 100`, `credit = 0`.
   - السطر الثاني: بدون عملة أجنبية أو بعملة الشركة، `credit` يساوي قيمة السطر الأول بعد التحويل.
3. اعرض القيد.

النتيجة المتوقعة:
- السطر الأجنبي يحفظ `debit_original = 100`.
- `debit` يحسب بعملة الشركة حسب `effective_rate`.
- `exchange_rate` و `rate_date` محفوظان في السطر.
- القيد لا يقبل الحفظ إذا لم يتوازن بعد التحويل.

## 10. إنشاء قيد بعملة أجنبية وسعر يدوي على السطر
الخطوات:
1. نفذ `POST /api/financial/entries` وأرسل في السطر الأجنبي `exchange_rate = 4.00000000` مع `debit_original = 100`.
2. اجعل السطر المقابل `credit = 400`.

النتيجة المتوقعة:
- النظام يستخدم سعر السطر المرسل بدلاً من السعر الفعلي الحالي.
- `debit = 400`.
- `exchange_rate = 4.00000000`.
- القيد متوازن ويمكن إنشاؤه كمسودة.

## 11. منع سطر يحتوي مدين ودائن معاً
الخطوات:
1. أنشئ قيداً فيه سطر يحتوي `debit > 0` و `credit > 0`.

النتيجة المتوقعة:
- يرجع خطأ: لا يمكن للسطر أن يحتوي مدين ودائن معاً.
- لا يتم إنشاء القيد.

## 12. منع سطر صفري
الخطوات:
1. أنشئ قيداً فيه سطر `debit = 0` و `credit = 0` بدون مبالغ أصلية.

النتيجة المتوقعة:
- يرجع خطأ: لا يمكن أن يكون السطر صفراً.

## 13. منع قيد غير متوازن بعد تحويل العملة
الخطوات:
1. أنشئ قيداً بسطر أجنبي ينتج `debit = 372`.
2. اجعل السطر المقابل `credit = 370`.

النتيجة المتوقعة:
- يرجع خطأ أن القيد غير متوازن.
- لا تحفظ أي سطور.

## 14. ترحيل قيد متعدد العملات
الخطوات:
1. أنشئ قيداً متوازناً يحتوي سطراً أجنبياً وسعراً محفوظاً.
2. نفذ `POST /api/financial/entries/{entry_ulid}/post`.
3. اعرض القيد وحسابات السطور.

النتيجة المتوقعة:
- ينتقل القيد إلى `posted`.
- لا تتغير مبالغ السطور بعد الترحيل.
- الحسابات في الدفاتر تعرض مبالغ الشركة الأساسية، مع بقاء تفاصيل العملة الأصلية في السطر.

## 15. دفتر الأستاذ للحساب
الخطوات:
1. أنشئ ورحّل قيداً يحتوي حساباً معيناً وسطر عملة أجنبية.
2. نفذ `GET /api/financial/accounts/{account_ulid}/ledger?fiscal_year_id={fiscal_year_ulid}`.

النتيجة المتوقعة:
- يظهر القيد في دفتر الأستاذ.
- الرصيد والمدينة/الدائنة بعملة الشركة الأساسية.
- عند عرض تفاصيل القيد تظهر العملة الأصلية وسعر الصرف.

## 16. ميزان المراجعة بعد قيود متعددة العملات
الخطوات:
1. رحّل قيداً واحداً على الأقل بعملة الشركة وقيداً آخر بعملة أجنبية.
2. نفذ `GET /api/financial/reports/trial-balance?fiscal_year_id={fiscal_year_ulid}`.

النتيجة المتوقعة:
- مجموع المدين يساوي مجموع الدائن.
- كل القيم محسوبة بعملة الشركة الأساسية.

## 17. مركز تكلفة مع عملة أجنبية
الخطوات:
1. اختر مركز تكلفة نشط وقابل للترحيل.
2. أنشئ قيداً بسطر أجنبي مرتبط بالمركز.
3. نفذ `GET /api/financial/cost-centers/{cost_center_ulid}/ledger?fiscal_year_id={fiscal_year_ulid}`.

النتيجة المتوقعة:
- يظهر أثر القيد على مركز التكلفة بعملة الشركة الأساسية.
- تفاصيل القيد تحتفظ بالعملة الأصلية وسعر الصرف.

## 18. فتح الصفحة الإدارية لأسعار الصرف
الخطوات:
1. افتح صفحة إدارة أسعار الصرف في لوحة الإدارة.
2. جرّب تغيير المصدر بين System Rate وCompany Rate وMarket API Rate.
3. جرّب البحث والفلترة حسب الأسعار الخاصة بالشركة.

النتيجة المتوقعة:
- التسميات واضحة وليست مختلطة بين العالمي والنظام والشركة.
- زر المزامنة يظهر فقط في Market API Rate.
- زر تعديل السعر يظهر فقط في Company Rate.

## 19. صفحة العملات الأساسية
الخطوات:
1. افتح صفحة العملات.
2. أنشئ عملة جديدة غير أساسية.
3. غيّر حالتها إلى غير نشطة.
4. تأكد من عدم ظهورها في dropdown العملات للقيود.

النتيجة المتوقعة:
- العملات غير النشطة لا تستخدم في العمليات المالية الجديدة.
- لا تتأثر القيود القديمة المحفوظة إن كانت مرتبطة بهذه العملة.

## 20. حدود الأمان والصلاحيات
الخطوات:
1. نفذ أي endpoint مالي بدون token.
2. نفذ endpoint مع `X-Company-Id` لشركة لا تخص المستخدم.
3. حاول استخدام حساب أو مركز تكلفة من شركة أخرى داخل قيد.

النتيجة المتوقعة:
- الطلب غير الموثق يرفض.
- بيانات الشركات لا تختلط.
- الحسابات ومراكز التكلفة والعملات الخاصة بشركة أخرى لا تُقبل.

## 21. سيناريو فشل API السوق
الخطوات:
1. احذف أو عطّل `OPEN_EXCHANGE_RATES_APP_ID` في بيئة اختبار.
2. اجعل المصدر `market`.
3. نفذ `POST /api/core/currencies/sync`.

النتيجة المتوقعة:
- يرجع خطأ واضح عن غياب API key.
- لا يتم إنشاء أسعار سوق ناقصة أو صفرية.
- تبقى الأسعار السابقة كما هي.

## 22. سيناريو تغيير العملة الأساسية للشركة
الخطوات:
1. غيّر `default_currency_id` للشركة إلى عملة أخرى في بيئة اختبار فقط.
2. اعرض قائمة العملات والأسعار.
3. أنشئ قيداً جديداً بعملة أجنبية.

النتيجة المتوقعة:
- العملة الأساسية الجديدة تعرض `effective_rate = 1`.
- كل الأسعار تعاد قراءتها نسبة إلى العملة الأساسية الجديدة.
- القيود الجديدة تحفظ قيمها بعملة الشركة الجديدة.

## 23. سيناريو تاريخ السعر
الخطوات:
1. أضف سعر شركة لعملة بتاريخ قديم.
2. أضف سعر شركة آخر لنفس العملة بتاريخ أحدث.
3. أنشئ قيداً بتاريخ كل فترة.

النتيجة المتوقعة:
- يجب مراجعة أن النظام يستخدم السعر المناسب حسب التاريخ. حالياً منطق القراءة يعتمد على آخر سعر ويقبل `date` في بعض المسارات، لذلك هذا السيناريو مهم لاكتشاف أي فجوة في اختيار السعر التاريخي.

## 24. سيناريو التقارير النهائية
الخطوات:
1. أنشئ قيود مبيعات/مصروفات بعملات متعددة.
2. رحّل القيود.
3. نفذ تقارير: دفتر الأستاذ، ميزان المراجعة، قائمة الدخل، الميزانية.

النتيجة المتوقعة:
- كل التقارير المالية تعرض أرقاماً متوازنة بعملة الشركة الأساسية.
- تفاصيل القيود تحفظ مرجع العملة الأصلية وسعر الصرف للمراجعة.

## ملاحظات QA
- احفظ ULID لكل شركة، سنة مالية، دفتر، حساب، عملة، ومركز تكلفة مستخدم في التجارب.
- لا تختبر تغييرات العملة الأساسية على بيانات إنتاجية.
- اختبر المصادر الثلاثة بالترتيب: System Rate، ثم Company Rate، ثم Market API Rate.
- بعد كل قيد متعدد العملات، افحص السطر نفسه لا التقرير فقط: `currency`, `debit_original`, `credit_original`, `exchange_rate`, `rate_date`.
